CARES

How to Give Helpful Answers

Whether you're helping someone in Discord, Stack Overflow, or your own community, the best answers solve the person's problem without overwhelming them.

Answer the question they asked.

Teach more only after you've solved the problem.

C — Correct

An incorrect answer is worse than no answer at all. If you're unsure, say so. It's okay to answer with "I think..." or let someone else respond.

✓ "This variable is out of scope because it was declared inside the if statement." ✗ "C++ randomly destroys variables after a while."

A — Actionable

Tell people what they can actually do. A good answer leaves the reader knowing exactly what their next step is.

✓ "Move the declaration above the loop." ✗ "Read more about variable lifetime."

R — Relevant

Answer the question that was asked—not the one you wish they'd asked. Avoid turning every help request into a full code review.

Question: "Why is this variable out of scope?" ✓ Explain scope. ✗ Critique formatting, naming, build system, logging framework, architecture, and coding style.

E — Efficient

The shortest complete answer is usually the best one. People can always ask follow-up questions if they want more detail.

✓ 3 concise paragraphs. ✗ A 2,000-word essay explaining every corner case.

S — Specific

Avoid vague advice. Point to the exact line, concept, or action that fixes the problem.

✓ "Line 42 should use '&' instead of copying the vector." ✗ "Something is probably wrong with your code."

A Common Mistake

Experienced developers often see dozens of opportunities to improve someone's code. That's great—but beginners usually only asked about one thing.

Question: "Why won't this compile?" Bad: • Use snake_case. • Don't use globals. • Rewrite this as ECS. • Use CMake. • Your architecture needs dependency injection. Better: "The compiler can't find this function because you forgot to include the header." (Optional) "Once that's working, there are a few style improvements if you're interested."

One More Thing

If someone has already answered, avoid repeating them just to speak. Only chime in if their answer is incorrect, incomplete, or needs crucial clarification.

The CARES Checklist

Before sending your message, ask yourself:

✔ Is it Correct?
✔ Is it Actionable?
✔ Is it Relevant?
✔ Is it Efficient?
✔ Is it Specific?
